Design Patterns

The computing scientist's main challenge is not to get confused by the complexities of his own making.

- E. W. Dijkstra
  • Common design problems as arising from "conflicting forces"
  • A design pattern is not invented it's discovered as a repeated solution to a commonly recurring problem
  • Is a formal way of documenting a solution to a design problem in a particular field of expertise
  • Decouple an abstraction from its implementation so that the two can vary independently


Each pattern describes a problem that occurs over and over again in our environment, and then describes the core of the solution to that problem, in such a way that you can use this solution a million times over, without ever doing it the same way twice.

--Christopher Alexander (UCB prof, father of PLM, 1st(?) hypertext book)

Classification of Design Patterns

Creational

  • Patterns that deal with object creation mechanisms; create objects in a manner suitable to the situation
  • Abstract Factory, Lazy Initialization, Singleton, etc

Structural

  • Patterns that ease the design by identifying a simple way to realize relationships between entities
  • Adapter, Decorator, Facade, etc

Behavioral

  • Patterns that identify common communication patterns between objects and realize these patterns
  • Command, Observer, Template Method, etc

Vocabulary of Design Patterns

Composition

  • Composition is a way to combine simple objects or data types into more complex ones
  • Is also known as a 'is a part of'

Aggregation

  • Aggregation differs from ordinary composition in that it does not imply ownership
  • Is also known as a 'has a' relationship

Delegation

Delegation refers to one object relying upon another to provide a specified set of functionalities.

  • Reduces the coupling of methods to their class
  • Easier to change components behavior in the future

(more) Vocabulary of Design Patterns

Cohesion

  • A measure of how strongly-related or focused the responsibilities of a single module are

Coupling

  • The degree to which modules rely on each other

Inheritance

  • A way to reuse code of existing objects, establish a subtype from an existing object

Interface

  • A tool and concept that refers to a point of interaction between components

Polymorphism

  • The ability to create a variable, a function, or an object that has more than one form

interrelated/trade offs, eg: less cohesion -> more coupling

Signals

Named signals are created with signal():

1 >>> from blinker import signal
2 >>> initialized = signal('initialized')
3 >>> initialized is signal('initialized')
4 True

Every call to signal('name') returns the same signal object (Singleton).

Subscribing to Signals

Signal.connect() registers a callback function.

1 >>> def subscriber(sender):
2 ...     print("Got a signal sent by %r" % sender)
3 ...
4 >>> ready = signal('ready')
5 >>> ready.connect(subscriber)

Emitting Signals

Producers call Signal.send() -> notifiy all receivers

 1 >>> class Processor:
 2 ...    def __init__(self, name):
 3 ...        self.name = name
 4 ...
 5 ...    def go(self):
 6 ...        ready = signal('ready')
 7 ...        ready.send(self)
 8 ...        print("Processing.")
 9 ...        complete = signal('complete')
10 ...        complete.send(self)
11 ...
12 ...    def __repr__(self):
13 ...        return '<Processor %s>' % self.name
14 ...
15 >>> processor_a = Processor('a')
16 >>> processor_a.go()
17 Got a signal sent by <Processor a>
18 Processing.

Notice the complete signal in go()?

No receivers have connected to complete yet, and that is ok.

Data Through Signals

Additional keyword arguments can be passed to send(). These will in turn be passed to the connected functions:

 1 >>> send_data = signal('send-data')
 2 >>> @send_data.connect
 3 ... def receive_data(sender, **kw):
 4 ...     print("Caught signal sent by %r, data %r" % (sender, kw))
 5 ...     return 'received!'
 6 ...
 7 >>> result = send_data.send(anonymous', abc=123)
 8 Caught signal sent by 'anonymous', data {'abc': 123}
 9 >>> result
10 [(<function receive_data at 0x...>, 'received!')]

Optimizing Signal Sending

If the data to be sent down a signal is very expensive, it can be more efficient to check to see if any receivers are connected first by testing the receivers property:

1 >>> bool(signal('ready').receivers)
2 True
3 >>> bool(signal('complete').receivers)
4 False

Checking for a receiver listening for a particular sender is also possible:

1 >>> signal('ready').has_receivers_for(processor_a)
2 True

Coda the 2nd: Weak References

Strong references are pointers to objects that have an effect on their reference counts

1 >>> a = [1,2,3]
2 >>> b = a
3 >>> # lots of operations involving a & b
4 >>> del a ; del b

When is a destoyed?

 1 >>> class Foo(object):
 2      def __init__(self):
 3        self.obj = None
 4        print 'created'
 5 
 6      def __del__(self):
 7          print 'destroyed'
 8 
 9      def show(self):
10          print self.obj
11 
12      def store(self, obj):
13          self.obj = obj
14 >>> a = Foo()
15 created
16 >>> b = a
17 >>> del a
18 >>> del b
19 destroyed

Weak References continued

Weak references have no effect on the reference count for an object

 1 import weakref
 2 >>> a = Foo()
 3 created
 4 >>> b = weakref.ref(a) # temp strong ref created by calling weak ref
 5 >>> a == b()
 6 True
 7 >>> b().show()
 8 None
 9 >>> del a
10 destroyed
11 >>> b() is None
12 True

Cyclic references

A need for weak references arises when objects have strong references forming a cycle:

 1 >>> a = Foo() ; b = Foo()
 2 created
 3 created
 4 >>> a.store(b) ; b.store(a)
 5 >>> del a ; del b
 6 >>> # Not destroyed!
 7 class Foo(object): 
 8     ...
 9     ...
10     def store(self, obj):
11         self.obj = weakref.ref(obj)
12 
13 >>> a = Foo() ; b = Foo()
14 created
15 created
16 >>> a.store(b) ; b.store(c)
17 >>> del a ; del b
18 destroyed
19 destroyed
